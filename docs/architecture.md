# Архитектура WeilShop

## 1. Концепция проекта

`WeilShop` - это `B2C`-сайт магазина автохимии на русском языке без онлайн-оплаты и доставки на первом этапе, но с:
- личным кабинетом
- системой заявок
- отслеживанием статусов заявок
- базой знаний
- административной панелью
- адаптивной мобильной версией
- заделом под Android-приложение

Рекомендуемый подход: `API-first`.

Система состоит из трех основных частей:
- `Frontend` витрины магазина
- `Backend API`
- `Admin Panel`

Такой подход позволяет:
- развивать сайт и админку независимо
- повторно использовать `API` для будущего Android-приложения
- упрощать SEO-оптимизацию публичной части

## 1.1. Дизайн-направление

Для `WeilShop` рекомендуется визуальная концепция `Tech Clean + Practical Expert`.

Это означает:
- чистый современный интерфейс
- технологичный, но не холодный визуальный язык
- акцент на сценарии ухода, инструкции и экспертную помощь
- приоритет понятности для новичка без потери ощущения качества

Сайт должен восприниматься не как шаблонный каталог, а как магазин-консультант по автохимии.

## 2. Рекомендуемый стек

- `Frontend`: `Next.js`
- `Backend`: `NestJS`
- `Admin Panel`: `Next.js` admin или `React Admin`
- `Database`: `PostgreSQL`
- `ORM`: `Prisma`
- `Cache / Sessions / Rate Limit`: `Redis`
- `Storage`: `S3-compatible storage`
- `Auth`: `JWT + Refresh Token`
- `Search`: сначала поиск на `PostgreSQL`, позже можно подключить `Meilisearch`

## 5. Ключевая бизнес-логика

### Каталог
Пользователь выбирает товары через:
- стандартный каталог
- категории
- бренды
- страницы задач ухода

### Задачи ухода
`CareTask` должна быть отдельной сущностью, а не обычным тегом. Примеры:
- мойка кузова
- очистка салона
- уход за пластиком
- очистка дисков
- уход за стеклами
- полировка
- защита кузова
- удаление сложных загрязнений

Для каждой задачи ухода нужны:
- описание проблемы
- рекомендации
- пошаговая инструкция
- FAQ
- список связанных товаров
- связанные статьи и инструкции
- готовые комплекты товаров по шагам

### Функции для удобства пользователя

Для `WeilShop` стоит заложить не только базовый каталог, но и сценарии, которые упрощают выбор автохимии:

- подбор по сценарию ухода
- пошаговые наборы товаров
- сравнение товаров
- быстрый подбор для новичка через мини-квиз
- отзывы с опытом применения
- FAQ в карточке товара
- связанные товары и аналоги
- избранное
- история просмотров
- повтор заявки
- уведомления о статусе заявки
- кнопка консультации по подбору
- предупреждения по совместимости и мерам предосторожности
- фильтр по уровню пользователя
- мобильные быстрые действия

### Приоритетные функции для первой полезной версии

Особенно рекомендуется включить в первую очередь:
- подбор по задачам ухода
- пошаговые комплекты товаров
- сравнение товаров
- FAQ и инструкции в карточке товара
- история статусов заявки с уведомлениями
- мини-квиз для подбора автохимии

### Заявки
На первом этапе вместо классического `checkout` используется `RequestOrder`.

Сценарий:
1. Пользователь добавляет товары в корзину.
2. Заполняет форму заявки.
3. Заявка отправляется администратору.
4. Администратор меняет статус заявки.
5. Пользователь видит актуальный статус в личном кабинете.

Рекомендуемые статусы:
- `new`
- `in_review`
- `confirmed`
- `processing`
- `completed`
- `cancelled`

### Регистрация и авторизация
Регистрация доступна:
- по `email`
- по номеру телефона

Поддерживается единая форма входа:
- логин: `email` или телефон
- пароль

Дополнительно:
- восстановление доступа
- подтверждение email
- архитектурная готовность к подтверждению телефона по SMS

## 8. Мобильная стратегия

Сайт должен проектироваться по принципу `mobile-first`.

Приоритеты:
- крупные интерактивные элементы
- быстрый доступ к каталогу и задачам ухода
- компактные фильтры
- удобная работа с карточками товаров
- быстрая отправка заявки
- фиксированные быстрые действия на карточке товара
- удобный квиз и выбор сценария ухода с мобильного

Это важно, так как в будущем планируется Android-приложение, которое сможет использовать тот же `API`.

## 9. Визуальные принципы интерфейса

- светлый, чистый фон без перегрузки
- графитовая типографика с уверенной иерархией
- глубокий зеленый как основной акцент бренда
- оранжевый как цвет действий и CTA
- страницы задач ухода визуально важнее обычных категорий
- карточка товара строится вокруг пользы и сценария применения
- база знаний оформляется как экспертный раздел, а не как вторичный блог

## 10. Нефункциональные требования

- адаптивность для мобильных устройств
- SEO-friendly архитектура
- безопасная авторизация
- масштабируемый backend
- расширяемость под мобильное приложение
- удобная админка для контент-менеджеров и администраторов

## 11. Архитектурные решения, которые стоит зафиксировать сразу

- корзина используется как основа формирования заявки
- `CareTask` является самостоятельной сущностью
- у заявки есть история смены статусов
- роли администраторов разделяются по правам
- backend проектируется как единая точка данных для сайта и будущего приложения
