# Безопасность WeilShop

Этот документ описывает политики и меры безопасности для проекта WeilShop.

---

## 1. Аутентификация и авторизация

### 1.1. Пароли

| Требование | Значение |
|------------|----------|
| Минимальная длина | 8 символов |
| Заглавные буквы | Обязательно |
| Строчные буквы | Обязательно |
| Цифры | Обязательно |
| Специальные символы | Опционально |
| Хеширование | bcrypt (cost factor 12) или argon2id |

**Запрещённые пароли:**
- `12345678`, `password`, `qwerty123`
- Email или телефон пользователя
- Имя или фамилия пользователя

### 1.2. JWT токены

| Параметр | Значение |
|----------|----------|
| Access token | 15 минут |
| Refresh token | 7 дней |
| Алгоритм подписи | HS256 |
| Хранение refresh token | HttpOnly cookie или Redis |

**Правила:**
- Access token хранится в памяти клиента
- Refresh token хранится в HttpOnly cookie с флагом `Secure` и `SameSite=Strict`
- При компрометации токенов — принудительный logout всех сессий
- Refresh token инвалидируется при смене пароля

### 1.3. Подтверждение аккаунта

- **Email:** обязательно подтверждение по ссылке с токеном (срок действия 24 часа)
- **Телефон:** архитектурная готовность к SMS-подтверждению (код действует 5 минут)

### 1.4. Восстановление доступа

- Токен сброса пароля действует 1 час
- Одноразовое использование токена
- После сброса пароля — инвалидация всех refresh токенов
- Rate limit: максимум 3 запроса в час на один email

## 2. Защита API

### 2.1. Rate Limiting

| Эндпоинт | Лимит | Окно |
|----------|-------|------|
| `/api/auth/*` | 10 запросов | 1 минута |
| `/api/requests` | 20 запросов | 1 минута |
| `/api/consultations` | 10 запросов | 1 минута |
| Остальные эндпоинты | 100 запросов | 1 минута |

**Превышение лимита:**
- Ответ: `429 Too Many Requests`
- Header: `Retry-After: 60`

### 2.2. CORS

```typescript
{
  origin: process.env.CORS_ORIGINS.split(','),
  credentials: true,
  methods: ['GET', 'POST', 'PATCH', 'DELETE', 'OPTIONS'],
  allowedHeaders: ['Content-Type', 'Authorization'],
  maxAge: 86400
}
```

### 2.3. Валидация входных данных

- Все входные данные валидируются через class-validator
- Строгая типизация через DTO
- Санитизация строк (XSS prevention)
- Максимальная длина полей ограничена

### 2.4. SQL Injection

- Использование Prisma ORM (параметризованные запросы)
- Запрет на raw queries без экстренной необходимости
- Code review для всех raw queries

## 3. Защита данных

### 3.1. Персональные данные (152-ФЗ)

**Хранение:**
- Пароли только в хешированном виде
- Email и телефон шифруются на уровне БД (опционально)
- Логи не содержат персональных данных

**Обработка:**
- Сбор только необходимых данных
- Явное согласие на обработку (checkbox в формах)
- Право на удаление данных по запросу пользователя

### 3.2. Чувствительные данные в логах

**Запрещено логировать:**
- Пароли и токены
- Полные номера карт (если появятся)
- Паспортные данные

**Маскирование:**
- Email: `i***@example.com`
- Телефон: `+7***0000000`

### 3.3. Резервное копирование

| Данные | Частота | Хранение |
|--------|---------|----------|
| PostgreSQL | Ежедневно | S3, 30 дней |
| Медиафайлы | Непрерывно | S3 с versioning |

## 4. Безопасность frontend

### 4.1. Security Headers

```http
Strict-Transport-Security: max-age=31536000; includeSubDomains
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
X-XSS-Protection: 1; mode=block
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self' data:; connect-src 'self' https://api.weilshop.ru
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: geolocation=(), microphone=(), camera=()
```

### 4.2. Защита от XSS

- Санитизация всего пользовательского контента
- React автоматически экранирует вывод
- Опасный HTML только через DOMPurify
- Content Security Policy (CSP)

### 4.3. Защита от CSRF

- Для mutation-операций требуется CSRF token
- Refresh token в HttpOnly cookie с SameSite=Strict
- Проверка Origin header для критических операций

## 5. Административная безопасность

### 5.1. Роли и доступ

| Роль | Создание админов | Изменение ролей | Доступ к данным |
|------|------------------|-----------------|-----------------|
| Super Admin | ✅ | ✅ | Полный |
| Catalog Manager | ❌ | ❌ | Каталог |
| Content Manager | ❌ | ❌ | Контент |
| Request Manager | ❌ | ❌ | Заявки |
| Consultation Manager | ❌ | ❌ | Консультации |
| SEO / Merch Manager | ❌ | ❌ | SEO, мерч |

### 5.2. Логирование действий администраторов

**Логируются:**
- Вход/выход из админки
- Изменение статусов заявок
- Создание/редактирование товаров
- Назначение ролей
- Экспорт данных

**Формат лога:**
```json
{
  "timestamp": "2025-01-15T14:30:00Z",
  "adminId": "uuid",
  "adminEmail": "admin@weilshop.ru",
  "action": "UPDATE_REQUEST_STATUS",
  "entity": "RequestOrder",
  "entityId": "uuid",
  "oldValue": "IN_REVIEW",
  "newValue": "CONFIRMED",
  "ipAddress": "192.168.1.1",
  "userAgent": "Mozilla/5.0..."
}
```

### 5.3. Сессии администраторов

- Сессия истекает через 8 часов без активности
- Принудительное завершение сессии при смене роли
- Максимум 3 активные сессии на администратора

## 6. Инфраструктурная безопасность

### 6.1. HTTPS

- Обязательно для продакшена
- TLS 1.3 минимум
- Автоматическое продление SSL-сертификатов

### 6.2. Сетевая безопасность

- Базы данных не доступны из публичной сети
- Доступ к админке только через VPN или whitelist IP (опционально)
- Разделение сетей: public, private, database

### 6.3. Зависимости

- Регулярный аудит через `npm audit` или `yarn audit`
- Обновление критических уязвимостей в течение 24 часов
- Использование lock-файлов (package-lock.json / yarn.lock)

## 7. Реагирование на инциденты

### 7.1. Классификация инцидентов

| Уровень | Описание | Время реакции |
|---------|----------|---------------|
| Critical | Утечка данных, компрометация аккаунтов | 1 час |
| High | DDoS, неработоспособность критических функций | 4 часа |
| Medium | Частичная неработоспособность | 24 часа |
| Low | Косметические проблемы, баги UI | 7 дней |

### 7.2. Процедура при утечке данных

1. Немедленная блокировка уязвимости
2. Сброс всех активных сессий
3. Уведомление затронутых пользователей
4. Уведомление регулятора (при необходимости по 152-ФЗ)
5. Постмортем и план предотвращения
